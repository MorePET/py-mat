# [PR 223](https://github.com/MorePET/mat/pull/223) test(visual): Bernhard's mat-vis #285 25-material grid + reviewer gate

## Summary

Comprehensive expansion of the visual-regression test for [mat-vis #285](https://github.com/MorePET/mat-vis/issues/285). v1 of this PR had a single visual test routed through `build123d.export_gltf` (which strips materials — separate gap not in scope here). v2 replaces it with **33 parametrized tests** across 5 classes that exercise py-mat's substrate → adapter → renderer pipeline directly.

## Five test classes

| Class | Tests | Pins |
|---|---|---|
| `TestBernhardMatVis285_AdapterStructure_Textured` | 10 (`to_threejs` / `to_gltf` × 5 tiers) | Adapter API drift + #285 scalar regression at every tier |
| `TestBernhardMatVis285_AdapterStructure_Scalar` | 10 (`to_threejs` / `to_gltf` × 5 physicallybased mats) | mat-vis #313 / py-mat #222 scalar-only path |
| `TestBernhardMatVis285_Ktx2Bytes` | 1 | KTX2 magic header at `tier="ktx2-1k"` (catches PNG-fallback regression) |
| `TestBernhardMatVis285_MtlxExport` | 1 | `.mtlx` XML parses + `<image file=…>` resolves on disk |
| `TestBernhardMatVis285_Visual` | 4 (3 textured tiers + 1 scalar) | End-to-end PNG via `to_threejs()` → JS Three.js (bypasses build123d) |

## Architecture change vs v1

**v1**: `build_scene → build123d.export_gltf → glTF on disk → Three.js GLTFLoader → screenshot` — lost materials at the build123d boundary.

**v2**: `to_threejs(vis) per material → JSON spec → JS-side MeshPhysicalMaterial constructor → Three.js scene → canvas.toDataURL` — no glTF middleware, no build123d dependency for the visual test. Texture data URIs fed directly into Three.js TextureLoader. Visual artifact reflects py-mat's actual surface only.

`tests/visual_grid_render.html` (new): programmatic Three.js viewer. Loads a JSON spec listing material spec dicts; instantiates one MeshPhysicalMaterial per spec; renders 5×5 grid of spheres. Uses Three.js's RoomEnvironment HDRI proxy + FOV-aware camera framing for full-scene visibility.
