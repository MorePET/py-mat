# [PR 219](https://github.com/MorePET/mat/pull/219) test(substrate): pin Bernhard's open mat-vis repros as strict xfails

## Summary

Adds `tests/test_substrate_repros.py` with three literal repros from the open mat-vis upstream issues Bernhard flagged in [build123d#1270](https://github.com/gumyr/build123d/pull/1270#issuecomment-4396673XXX):

| Test | Pins | Upstream issue |
|---|---|---|
| `TestMatVis285_GpuopenBakingScalars` | Two distinct gpuopen metals must NOT both return baker-default scalars (`metalness=0.0`, `roughness=0.5`, `color=0xCCCCCC`) | [mat-vis #285](https://github.com/MorePET/mat-vis/issues/285) |
| `TestMatVis311_MaterialNames` | `client.materials('gpuopen', '1k')` must not return bare UUIDs | [mat-vis #311](https://github.com/MorePET/mat-vis/issues/311) |
| `TestMatVis313_PhysicallybasedAccess` | `Vis('physicallybased', 'Aluminum').to_threejs()` must not raise `MaterialNotStagedError` | [mat-vis #313](https://github.com/MorePET/mat-vis/issues/313) |

Each test is a strict `xfail` today (the bugs are open upstream); when `mat-vis-client` ships the fixes and py-mat picks up the new dep version, the tests `XPASS` and the strict marker prompts dropping the xfail.

## Why

Closes the verification gap the build123d#1270 thread surfaced: running py-mat's own test suite or Bernhard's `tests/test_materials.py` passed cleanly even while these substrate paths were broken. The literal repro snippets from filed issues are the cheapest way to catch this class of "unasserted contract on a covered codepath."

## Network dependency

Hits real mat-vis substrate. Skip with `MAT_VIS_SKIP_LIVE=1` (mirrors the [`test_e2e_vis.py`](https://github.com/MorePET/mat/blob/main/tests/test_e2e_vis.py) convention). Transient CDN failures turn into `pytest.skip` via the local `_skip_on_upstream_outage` context manager.

## Out of scope

[mat-vis #312](https://github.com/MorePET/mat-vis/issues/312) (download progress) deliberately omitted: testing progress-callback behavior requires callback-capture infrastructure that doesn't exist on either side yet. Worth a separate issue once the callback API is designed.
