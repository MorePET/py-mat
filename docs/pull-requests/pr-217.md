# [PR 217](https://github.com/MorePET/mat/pull/217) test: xfail-pin to_threejs color as hex string (py-mat #99 / mat-vis #298)

## Summary

Adds a strict-xfail test asserting `to_threejs(m)["color"]` is a Pythonic `'#RRGGBB'` hex string (not the Three.js wire-format hex int we currently emit, e.g. `12566468` for `#bfbfc4`).

Closes the test-coverage gap from py-mat #99 (Bernhard's original report). Routed upstream as [mat-vis #298](https://github.com/MorePET/mat-vis/issues/298) with a concrete `color_format=Literal["hex","int","tuple"]` proposal defaulting to `"hex"`.

## Why a strict xfail

- **Documents the contract** — when a future reader sees the test, they know the desired output shape without reading the upstream issue chain.
- **Catches the upstream fix automatically** — `py-materials` consumes `mat-vis-client>=0.6.3`; the next 0.7.x with #298 default-flips this test to passing without code changes downstream. `strict=True` then triggers `XPASS` failure → reminder to drop the xfail mark.
- **Blocks regression in either direction** — if upstream goes back to int after fixing, we notice.
